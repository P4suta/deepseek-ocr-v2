# 仕組みと知見

このプロジェクトで得られた、再現性のある技術的知見をまとめます。

## flash-attn は不要

DeepSeek-OCR-2 のカスタムモデルコードを直接確認した結果:

- 視覚エンコーダ（`deepencoderv2.py`）は flash_attn を import せず、`sdpa`/`eager` を要求（むしろ `flash_attention_2` を拒否）。
- デコーダ（`modeling_deepseekv2.py`）は flash_attn を条件付き import。`ATTENTION_CLASSES` に `eager` がある（ただし `sdpa` キーは無い）。

→ **`_attn_implementation="eager"` を渡せば、flash-attn 未インストールでもロード・実行できる**。Windowsでの flash-attn ビルドを回避できる。

## 6GB GPU での bf16 vs 4bit

3B（bf16で約6.3GB）は専用VRAM 6GBを少し超えるが、NVIDIA の **CUDA Sysmem Fallback**（標準ON）で共有メモリへ退避し動作する。実測:

| | ロード | 1ページ推論 |
|---|---|---|
| 4bit | 16s | 32.8s |
| bf16 | 17s | 26.9s |

bf16 は退避が約1GBと軽微で、4bit には逆量子化のオーバーヘッドがあるため、**bf16でも4bitより1文字あたり約12%遅いだけ**。精度はbf16が上。「bf16は遅すぎる」は誤りだった。

## 縦書き日本語の崩壊（degeneration）と解決

密な縦書きページは途中から同じ語句を繰り返して崩壊する。検証で分かったこと:

- **画質や量子化の問題ではない**（bf16でも4bitでも同じ位置で崩壊、元画像は既に二値で綺麗）。
- 真因は **低解像度**。3496pxの画像を1024に潰すと文字が小さくなりすぎて読めず崩壊する。
- **解決**: PDFに埋め込まれた**ネイティブ高解像度画像**を抽出し、**`gundam`（crop_mode=True）でタイル分割**して渡す → 各タイルで文字が大きく保たれ、ページ全文を崩壊なく読み切れる。

!!! tip "試したが不採用だったもの"
    - **列ごとに分割して渡す**: 網羅性は上がるが、LLMが全体文脈で補完できず疎な列で幻覚する（古典OCR的でLLMと相性が悪い）。
    - **repetition_penalty**: 崩壊は減るが幻覚的な続きを生み精度が下がる（既定OFF）。
    - **画像の二値化・シャープ化**: 元が二値なので無意味。

## 無限反復の機械的カット（`repetition.py`）

- **生成中の停止**: `RepetitionStoppingCriteria` が「同じ n-gram が規定回数以上出現」を検知して `generate` を停止。短い周期反復から文単位の長いループまで捕捉。`.infer()` が内部で固定引数の `generate` を呼ぶため、`model.generate` をラップして注入する。
- **後処理**: `cleanup.py` で句の即時反復・隣接重複文を圧縮。

## 汎用クリーンアップの設計（`cleanup.py`）

特定の本に依存しないため、次の汎用ロジックを採用:

- **空行ブロック単位で分類**（見出し / リスト / 表 / 本文）。
- **柱・ノンブルの除去**: 純数字行、行内ノンブル（CJKに隣接するアラビア数字）、繰り返す短い見出し（=柱）は初出のみ採用し重複は破棄。
- **見出し推定**: 短く・文末記号で終わらない単一行を `##` に。前後のノンブルを正規化して重複排除。
- **本文のリフロー**: ページを跨いで連結。節番号 `[14]`「2」は脚注参照 `[^14]` として保持しつつ段落の区切りに。
- **崩壊ページ検出**: 句反復を畳んでも短すぎ／高反復なページを検出して印付け（自動削除しない）。

!!! warning "ヒューリスティックの限界"
    - 見出し推定は誤検出しうる（柱の誤OCRや短い断片を拾うことがある）。
    - 真の段落境界（字下げ）はOCR出力に残らないため、見出し＋節番号での粗い段落化に留まる。
    - 文字レベルのOCR誤りは対象外（モデル側の課題）。
    - 文字多様性（distinct ratio）による崩壊判定は、日本語では仮名多用で正常文も低値になり誤検出するため不採用。

## 環境メモ

- `grep`/`wc -m` はCロケールで多バイト文字を誤判定する（バイト数を数える・正規表現の `^$` が空振りする）。検証は Python で行うのが確実。
- Windowsの標準出力（パイプ）は cp932 になり日本語の print で落ちることがある。`PYTHONUTF8=1` を付けて実行する。
